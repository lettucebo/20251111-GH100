## 回報安全漏洞

**請勿透過公開的 Issue 回報安全漏洞。**

如果您發現安全漏洞，我們非常感謝您負責任地向我們揭露。請遵循以下步驟：

### 1. 聯絡方式

請透過以下任一方式與我們聯繫：

- **建議方式**：使用 GitHub 的私密漏洞回報功能（Private Vulnerability Reporting）
  - 前往本專案的 "Security" 標籤
  - 點擊 "Report a vulnerability"
  - 填寫漏洞詳細資訊

- **備用方式**：透過 Issue 聯絡專案維護者
  - 請注意：僅在無法使用私密回報時使用此方式
  - 切勿在 Issue 中揭露漏洞細節

### 2. 回報內容

請在回報中包含以下資訊：

- **漏洞類型**：例如 SQL 注入、跨站腳本攻擊（XSS）、權限提升等
- **影響範圍**：受影響的檔案、功能或元件
- **重現步驟**：詳細的步驟說明，讓我們能夠重現問題
- **潛在影響**：說明此漏洞可能造成的安全風險
- **建議修復方案**（選用）：如果您有修復建議，我們非常歡迎
- **環境資訊**：作業系統、瀏覽器版本、相關軟體版本等

### 3. 範例格式

```
漏洞類型：[漏洞類型]
影響範圍：[檔案/功能/元件]

重現步驟：
1. [步驟 1]
2. [步驟 2]
3. [步驟 3]

潛在影響：
[描述可能的安全風險]

環境資訊：
- 作業系統：[例如 Windows 11, macOS 14]
- 瀏覽器：[例如 Chrome 120]
- 其他相關版本資訊

建議修復方案（選用）：
[您的建議]
```

## 安全漏洞揭露流程

我們遵循負責任的漏洞揭露原則：

### 我們的承諾

1. **及時回應**：我們會在收到回報後 3 個工作天內確認收到您的訊息
2. **保持溝通**：我們會定期更新調查進度
3. **致謝回饋**：在修復完成後，我們會在公告中感謝您的貢獻（除非您希望匿名）
4. **協同修復**：我們歡迎與回報者合作，共同制定修復方案

### 您的承諾

為了確保安全漏洞得到妥善處理，我們希望您：

1. **保密性**：在問題修復並公開發布前，請勿公開揭露漏洞細節
2. **合理測試**：請勿在未經授權的情況下，對實際營運系統進行測試
3. **善意行為**：僅以發現和回報漏洞為目的進行測試
4. **尊重隱私**：請勿存取、修改或刪除其他使用者的資料

## 回應時程

我們致力於快速處理安全問題，以下是我們的目標時程：

| 階段 | 目標時程 |
| --- | --- |
| 初步確認回報 | 3 個工作天 |
| 問題評估與分析 | 7 個工作天 |
| 修復方案開發 | 根據嚴重程度而定 |
| 安全更新發布 | 根據嚴重程度而定 |

### 嚴重程度分級

- **嚴重（Critical）**：立即著手處理，目標 24-48 小時內發布修復
- **高（High）**：優先處理，目標 1-2 週內發布修復
- **中（Medium）**：正常處理，目標 2-4 週內發布修復
- **低（Low）**：依排程處理，目標 1-3 個月內發布修復

實際時程可能因問題複雜度而有所調整，我們會在評估後告知您預估的修復時程。

## 安全最佳實踐

### 對於使用者

如果您正在使用本專案，建議您：

1. **定期更新**：保持使用最新版本
2. **檢視變更日誌**：關注安全相關的更新
3. **安全設定**：遵循專案文件中的安全設定建議
4. **環境隔離**：在開發環境中測試新版本後再部署到正式環境

### 對於貢獻者

如果您正在為本專案做出貢獻，請注意：

1. **安全審查**：在提交 PR 前檢查是否有潛在的安全問題
2. **敏感資訊**：切勿在程式碼中包含密碼、金鑰或其他敏感資訊
3. **依賴套件**：定期更新依賴套件，並檢查是否有已知漏洞
4. **輸入驗證**：始終驗證和清理使用者輸入
5. **遵循規範**：參考 [CONTRIBUTING.md](CONTRIBUTING.md) 了解貢獻規範
