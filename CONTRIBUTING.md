## 行為準則

參與此專案的所有貢獻者都應遵守以下基本原則：

- 尊重所有參與者，保持友善和專業的態度
- 接受建設性的批評和回饋
- 專注於對社群最有利的事情
- 展現同理心和包容性

## 如何貢獻

### 回報問題

如果您發現錯誤或有功能建議，請遵循以下步驟：

1. **搜尋現有的 Issues**
   - 在建立新 Issue 之前，請先搜尋是否已有類似的問題
   - 如果找到相關的 Issue，可以在該 Issue 下留言補充資訊

2. **建立新的 Issue**
   - 使用清晰、具描述性的標題
   - 提供詳細的問題描述，包括：
     - 預期行為
     - 實際發生的情況
     - 重現步驟
     - 相關的錯誤訊息或截圖
     - 您的環境資訊（作業系統、瀏覽器等）

### 提交程式碼

#### 開發環境設定

1. **Fork 此專案**
   
   點擊 GitHub 頁面右上角的 "Fork" 按鈕

2. **Clone 您的 Fork**
   ```bash
   git clone https://github.com/您的使用者名稱/20251111-GH100.git
   cd 20251111-GH100
   ```

3. **設定上游遠端倉庫**
   ```bash
   git remote add upstream https://github.com/lettucebo/20251111-GH100.git
   ```

4. **保持您的 Fork 同步**
   ```bash
   git fetch upstream
   git checkout main
   git merge upstream/main
   ```

#### 建立功能分支

為您的變更建立一個新分支：

```bash
git checkout -b feature/your-feature-name
```

或針對錯誤修復：

```bash
git checkout -b fix/your-bug-fix
```

分支命名建議：
- `feature/` - 新功能
- `fix/` - 錯誤修復
- `docs/` - 文件更新
- `refactor/` - 程式碼重構
- `test/` - 測試相關

## 提交 Pull Request

1. **確保您的程式碼品質**
   - 程式碼遵循專案的編碼規範
   - 所有測試都通過
   - 新增適當的測試來驗證您的變更

2. **Commit 您的變更**
   ```bash
   git add .
   git commit -m "描述您的變更"
   ```

3. **推送到您的 Fork**
   ```bash
   git push origin feature/your-feature-name
   ```

4. **建立 Pull Request**
   - 前往 GitHub 上的原始專案
   - 點擊 "New Pull Request"
   - 選擇您的分支
   - 填寫 PR 描述，包括：
     - 變更的目的和內容
     - 相關的 Issue 編號（如果有）
     - 測試方式
     - 截圖（如果是 UI 變更）

5. **等待審核**
   - 維護者會審核您的 PR
   - 可能會要求您進行修改
   - 請及時回應審核意見

## 程式碼規範

### 一般原則

- 保持程式碼簡潔、可讀
- 適當的註解說明複雜邏輯
- 遵循 DRY（Don't Repeat Yourself）原則
- 確保向後相容性

### 文件規範

- 使用繁體中文撰寫文件
- 保持格式一致性
- 提供清晰的範例
- 更新相關的說明文件

## Commit 訊息規範

### 格式

```
<類型>: <簡短描述>

<詳細說明>（可選）

<相關 Issue>（可選）
```

### 類型

- `feat`: 新功能
- `fix`: 錯誤修復
- `docs`: 文件更新
- `style`: 程式碼格式調整（不影響功能）
- `refactor`: 程式碼重構
- `test`: 測試相關
- `chore`: 建置流程或輔助工具的變動

### 範例

```
feat: 新增課程範例程式碼

新增 GitHub Actions 工作流程範例，展示 CI/CD 基本概念

關閉 #123
```

```
fix: 修正 README 中的連結錯誤

修正文件中失效的外部連結
```

```
docs: 更新貢獻指南

新增更詳細的 PR 提交流程說明
```

## 問題解答

### 我的 PR 需要多久才會被審核？

我們會盡快審核所有的 PR，通常在幾天內會有回應。如果超過一週沒有回應，歡迎在 PR 中留言提醒。

### 我可以同時處理多個 Issue 嗎？

可以，但建議一次專注於一個 Issue，並為每個 Issue 建立獨立的分支和 PR。

### 我不確定如何開始，該怎麼辦？

歡迎在 Issue 中提問，或尋找標記為 `good first issue` 的問題開始。
